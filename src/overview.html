The Rapt library is a compendium of very small tools sharing a common theme of the apt tool new in J2SE 5.0.

Tools in the Rapt libray have the following lifecycle.

Experimental
These tools are experimental, they are being evaluated to see firstly if something is possible, and secondly whether it is feasible.
Experimental tools are not shipped in the binary download, and are only available in source code form. (We might change our minds about that and ship a separate experimental binary. Experimental tools are volatile, and no reliance on them should be made in production code.
Proposed
Proposed tools have their utility accepted (that is, we think they are useful), and are close to their final form. We really like feedback on proposed tools, and the javadocs should contain a link to a discussion forum where you can comment on them. In some cases we may have some undecided features, we will list these in the javadoc, and we would like to know your views.
If we change the functionality of a proposed tool, we will make sure it is backward compatible, so your existing code will still work. We will also leave the proposed version as is when we release the version we are happy with. If necessary we will achieve this by issuing a Mark II proposed version in a different package from the newest version that is compatible with the original proposed version. IF we ever do this, we will decide what the package is called and let you know.
Released
Released tools are stable and functionally complete in our opinion.

The life cycle stage, is represented in the package names

  • net.java.dev.rapt.experimental...

  • net.java.dev.rapt.proposed...

  • net.java.dev.rapt.released...

The rapt library consists of

Generators
These are annotations and associated processors which work by automatically generating additional source code (when you source is compiled with apt instead of javac).
Mirror tools
These are utility classes helpful to those writing their own annotations and processors.

These are represented as the package name below the life cycle. eg net.java.dev.rapt.proposed.generators

Using Generators

Each generator annotation will contain details of what it does and how to use its API description.

Once you have written some code which uses a generator you need to do the following.

  1. Compile your code with J2SE 5.0 (or later) "apt" tool rather than javac.

  2. Pass apt all the options you would normally pass to javac to build your code.

  3. Always specify a directory to place the generated source code in using apt's -s option. (if you don't it generates it in the same directory as the source, which works the first time, but the next time you compile *.java, it compiles the generated source on the first round, then when it compiles the newly generated source in the second round, there is a duplicate class error.

  4. Always specify a destination directory with -d, (even if you dont with javac) otherwise the generated source's class files will be placed in the -s directory, rather than to the same place as your other class files. If the destination is the current directory use -d .

  5. make sure rapt.jar is on your classpath.

  6. do NOT use apt's -factory option, it will find the annotation Processors automatically.

Once you have compiled (successfully) with apt, the generator should work as documented. You may view the generated source code if you are interested, it will be under the directory specified by the -s option.



Unless stated, in the Annotation Type API, rapt's processors do not need any -A options on the apt command line.
